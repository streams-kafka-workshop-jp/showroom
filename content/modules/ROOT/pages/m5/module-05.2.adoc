== Schema Registryを使用したスキーマ取得、メッセージの送受信

このセクションでは、前のセクションでSchema Registryに追加したスキーマを使用するProducerとConsumerを作成します。

* link:https://devspaces.{openshift_cluster_subdomain}[Devspaces IDE, window="devspaces"] に移動します

* 必要に応じて認証情報 - `{user_name}` と `{password}` でログインします。

* Devspacesダッシュボードから、*open* ボタンをクリックしてワークスペースを起動します。
+
image::m2/devspaces-launch.png[]

* ファイルエクスプローラーで *module-registry/src/main/resources/application.properties* ファイルを開きます
+
image::m5/registry-app-prop-placeholder.png[]

* プレースホルダーの値を以下のregistry API URLに置き換えます
+
[source,URL,role="execute",subs=attributes+]
----
https://service-registry-app-{user_name}.{openshift_cluster_subdomain}/apis/registry/v3
----

* bootstrap serversのプレースホルダーの値を以下の値に置き換えます
+
[source,bash,role="execute",subs=attributes+]
----
kafka-kafka-bootstrap.kafka-{user_name}.svc.cluster.local:9092
----


* 設定ファイルは以下のようになります。registry API URLとbootstrap serversに加えて、このファイルに必要なgroup id、artifact id、versionトピック名、consumer group情報も定義されていることに注目してください。
+
image::m5/registry-app-prop.png[]

* 左上のメニューをクリックして新しいターミナルを起動します。 *Terminal > New Terminal.* を選択して新しいターミナルウィンドウを開きます。
+
image::m5/new-terminal.png[]

* ターミナルでmodule-registryディレクトリに移動します
+
[source,sh,role="execute",subs=attributes+]
----
cd module-registry
----

// * In the terminal window, run the following command to make sure you are in the project directory. It should print the current working directory as `/projects/registry-java`.
// +
// [source,sh,role="execute",subs=attributes+]
// ----
// pwd
// ----

* ターミナルがJava 21を使用するように設定されていることを確認します。
+
[source,sh,role="execute",subs=attributes+]
----
sdk install java 21.0.2-tem && sdk use java 21.0.2-tem
----

* 以下のコマンドを使用して `schema-demo` という名前のトピックを作成します。前のセクションでスキーマを作成する際に、artifact idと同じ名前を使用したことを思い出してください。 
+
[source,bash,role="execute",subs=attributes+]
----
oc run -n kafka-{user_name} topic-create --rm --restart=Never --quiet --attach \
--image=registry.redhat.io/amq-streams/kafka-40-rhel9:3.0.0 -- \
bash -lc '/opt/kafka/bin/kafka-topics.sh --bootstrap-server kafka-kafka-bootstrap:9092 --create --topic schema-demo --partitions 3 --replication-factor 1'
----


* 以下のコマンドを実行してアプリケーションをビルドし、インタラクティブアプリケーションを実行します
+
[source,sh,role="execute",subs=attributes+]
----
./run.sh
----
* 以下のオプションを持つインタラクティブアプリケーションが起動するはずです。
+
[source,bash]
----
======================================
Apicurio Registry Producer Demo
======================================
1. Send custom message (JSON input)
2. Fetch and display schema from registry
3. Consume messages from topic
4. Exit
======================================
Select scenario (1-4):
----

=== スキーマ取得
* まず、Schema Registryからスキーマを取得しましょう。`2` を入力してオプション2を選択し、Enterキーを押します。

* 以下のような出力が表示され、スキーマがSchema Registryから正常に取得されたことが示されます。前のセクションで追加したのと同じスキーマが表示されるはずです。
+
image::m5/retrieve-schema-output.png[]


=== スキーマに準拠したメッセージの送信

* 次に、Producerを実行してメッセージを送信しましょう。`1` を入力してオプション1を選択し、Enterキーを押します。

* 以下のような出力が表示され、メッセージの入力を求められます。
+
[source,bash]
----
--- Enter your JSON message (single-line or multi-line, press Enter on empty line when done): ---
----

* このメッセージをターミナルに貼り付けて、 *Enterキーを2回押します*。このJSONは有効で、schema registryで定義されたスキーマに準拠しているため、受け入れられて `schema-demo` トピックに送信されるはずです。
+
[source,json,role="execute",subs=attributes+]
----
{
  "customer_id": "CUST-001",
  "first_name": "John",
  "last_name": "Doe",
  "email": "john.doe@example.com",
  "age": 30,
  "is_active": true,
  "signup_date": 1699123456789
}
----

* 以下のような出力が表示され、メッセージが正常に送信されたことが示されます。これは、Producerがschema registryからスキーマを取得し、スキーマに対してメッセージを検証し、`schema-demo` トピックにメッセージを送信できたことを示しています。
+
[source,bash]
----
✓ Message sent successfully!
   Validation: ✓ Automatic (by Apicurio Registry)
   Schema source: Registry (artifact 'demo' in group 'my-group')
   Record sent: {"customer_id": "CUST-001", "first_name": "John", "last_name": "Doe", "email": "john.doe@example.com", "age": 30, "is_active": true, "signup_date": 1699123456789}
----

* link:https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}/kafka[streams console, window="console"] をクリックしてstreams consoleに移動し、*schema-demo* トピックに移動して、新しいメッセージがあることを確認します。

=== スキーマに準拠しないメッセージの送信
* 次に、無効なメッセージを送信してみましょう。`1` を入力してオプション1を選択し、Enterキーを押します。

* 以下のような出力が表示され、メッセージの入力を求められます。  
+
[source,bash]
----
--- Enter your JSON message (single-line or multi-line, press Enter on empty line when done): ---
----

* このメッセージをターミナルに貼り付けて、 *Enterキーを2回押します*。このJSONは無効です。なぜなら、スキーマでは `customer_id` の型が `string` であることを示していますが、これは整数値を提供しているからです。このメッセージは受け入れられないはずです。
+
[source,json,role="execute",subs=attributes+]
----
{
  "customer_id": 12345,
  "first_name": "John",
  "last_name": "Doe",
  "email": "john.doe@example.com",
  "age": 30,
  "is_active": true,
  "signup_date": 1699123456789
}
----

* アプリケーションがエラーをスローするのが確認でき、エラーログをスクロールすると、以下のエラーメッセージが表示されるはずです。
+
[source,bash]
----
Caused by: java.lang.ClassCastException: value 12345 (a java.lang.Long) cannot be cast to expected type string at Customer.customer_id
----

* streams consoleに戻ると、新しいメッセージが送信されなかったことを確認できます。

=== Schema Registryの互換性ルールの操作

* link:https://service-registry-ui-{user_name}.{openshift_cluster_subdomain}/explore/my-group/schema-demo[service registry UI, window="_schema-registry"] > my-group > schema-demoに移動します

* rulesタブに移動し、 *Compatibility rule* 行の *Enable* ボタンをクリックします。
+
image::m5/enable-rule.png[]


* ドロップダウンから `backward` オプションを選択します。このオプションは、スキーマの将来のバージョンの後方互換性をチェックします。
+
image::m5/backward.png[]

* Overviewタブをクリックし、 *Create version* ボタンをクリックします。
+
image::m5/create-version.png[]

* 既存のスキーマに `phone number` という新しいフィールドを導入し、それを必須フィールドにする新しいバージョンのスキーマを貼り付けます。これは後方互換性を破り、registryの互換性ルールをトリガーするはずです。実際に見てみましょう。

* 以下のスキーマをエディタにコピー&ペーストします。
+
[source,avro,role="execute",subs=attributes+]
----
{
    "type": "record",
    "name": "Customer",
    "namespace": "com.example.avro",
    "fields": [
      {"name": "customer_id", "type": "string"},
      {"name": "first_name", "type": "string"},
      {"name": "last_name", "type": "string"},
      {"name": "email", "type": ["null", "string"], "default": null},
      {"name": "phone", "type": "long"},
      {"name": "age", "type": ["null", "int"], "default": null},
      {"name": "is_active", "type": "boolean", "default": true},
      {"name": "signup_date", "type": "long"}
    ]
  }
----

* 以下のようなエラーメッセージが表示されるはずです。
+
image::m5/error-compatibility.png[]

* これは、新しいスキーマが既存のスキーマと互換性がないことを示しており、schema registryが後方互換性チェックを行い、新しいスキーマが既存のスキーマと互換性があることを保証しています。

* エラーメッセージダイアログを閉じます。

* *Create Version* ボタンを再度クリックし、既存のスキーマを以下のスキーマに置き換えます。これは先ほど提供したものと同じですが、`phone` フィールドが欠落している場合のデフォルト値を提供します。今回は互換性ルールがトリガーされず、新しいバージョンのスキーマが作成されるはずです。
+
[source,avro,role="execute",subs=attributes+]
----
{
    "type": "record",
    "name": "Customer",
    "namespace": "com.example.avro",
    "fields": [
      {"name": "customer_id", "type": "string"},
      {"name": "first_name", "type": "string"},
      {"name": "last_name", "type": "string"},
      {"name": "email", "type": ["null", "string"], "default": null},
      {"name": "phone", "type": "long", "default": 0},
      {"name": "age", "type": ["null", "int"], "default": null},
      {"name": "is_active", "type": "boolean", "default": true},
      {"name": "signup_date", "type": "long"}
    ]
  }
----

* これで、Producerは `phone` フィールドの有無に関わらずメッセージを送信でき、メッセージは依然として受け入れられます。これにより、スキーマが後方互換性を持つようになります。

// * Navigate back to dev spaces terminal and run the below command to start the application
// +
// [source,bash,role="execute",subs=attributes+]
// ----
// ./run.sh
// ----

* オプション1を選択し、以下のメッセージを入力します。Enterキーを2回押します。
+
[source,json,role="execute",subs=attributes+]
----
{
  "customer_id": "CUST-033",
  "first_name": "Jane",
  "last_name": "Doe",
  "age": 42,
  "is_active": true,
  "signup_date": 1699123456789
}
----

* `phone` フィールドのないメッセージが受け入れられ、`schema-demo` トピックに送信されます。

* `phone` フィールドを含めずに送信したメッセージについて、Consumerは後方互換性を確保するために `phone` フィールドにデフォルト値の0を使用することに注目してください。

* Producerを再度実行し、オプション1を選択して以下のメッセージを入力します。Enterキーを2回押します。
+
[source,json,role="execute",subs=attributes+]
----
{
  "customer_id": "CUST-035",
  "first_name": "Richard",
  "last_name": "Roe",
  "age": 52,
  "phone": 4443456789,
  "is_active": true,
  "signup_date": 1699123456789
}
----

* 有効な `phone` フィールドを持つこのメッセージも受け入れられ、`schema-demo` トピックに送信されます。これは、schema registryの後方互換性機能を示しており、新しいバージョンのスキーマが導入されても、古いバージョンのスキーマを使用しているProducerがメッセージを送信できるようにします。

* 次に、オプション3を選択してConsumerを実行し、新しいメッセージを確認します。 

* 以下のような出力が表示され、メッセージが正常にconsumeされたことが示されます。
+
[source,bash]
----
✓ Message received and deserialized:
  Key: custom-key
  Value: {"customer_id": "CUST-001", "first_name": "John", "last_name": "Doe", "email": "john.doe@example.com", "age": 30, "is_active": true, "signup_date": 1699123456789}
  Partition: 0
  Offset: 0

[main] INFO com.example.KafkaConsumerService - Received message - Key: custom-key, Partition: 0, Offset: 1
✓ Message received and deserialized:
  Key: custom-key
  Value: {"customer_id": "CUST-033", "first_name": "Jane", "last_name": "Doe", "email": null, "phone": 0, "age": 42, "is_active": true, "signup_date": 1699123456789}
  Partition: 0
  Offset: 1

[main] INFO com.example.KafkaConsumerService - Received message - Key: custom-key, Partition: 0, Offset: 2
✓ Message received and deserialized:
  Key: custom-key
  Value: {"customer_id": "CUST-035", "first_name": "Richard", "last_name": "Roe", "email": null, "phone": 4443456789, "age": 52, "is_active": true, "signup_date": 1699123456789}
  Partition: 0
  Offset: 2
----

* *'ctrl+c'* を押してConsumerアプリケーションを停止します。

* Consumerはschema registryからスキーマを取得し、メッセージを正常にdeserializeできます。これが、link:https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}/kafka[streams console, window="_amqstreams"] から直接ではなく、Consumer出力でメッセージを確認できる理由です。



=== まとめ

これでこのモジュールは終了です。さまざまな互換性ルールについてさらに理解を深めるには、link:https://docs.redhat.com/en/documentation/red_hat_build_of_apicurio_registry/3.1/html/apicurio_registry_user_guide/registry-rule-reference_registry#registry-rule-types_registry[こちらのドキュメント, window="docs"] を参照してください。

Schema registryは、データストリーミングアプリケーションのスキーマを管理できる強力なツールです。スキーマが相互に互換性を持つことを保証し、新しいスキーマが既存のスキーマと互換性があることを確認し、ニーズに合わせた複数の互換性ルールを提供します。
