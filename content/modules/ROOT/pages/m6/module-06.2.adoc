== Access Control Lists (ACLs)
*Access Control Lists (ACLs)* は、Kafkaクラスタへのアクセスを細かく調整するために使用されます。各IDがアクセスできる内容を制御できます。Access Control Lists (ACLs)は細かいレベルで定義されます。例えば、ユーザーにTopicからの読み取りは許可するが、書き込みや削除は許可しない、といったことが可能です。

ACLはクライアントアプリケーションにアクセスルールを適用します。ACLは、認証されたクライアントのIDが検証された後に何ができるかを決定します。

Authorizationは、{streams}に付属するStandardAuthorizerプラグインを使用して実装されます。ACL authorizerはACLルールを使用してKafka Brokerへのアクセスを管理します。ACLルールは以下の形式で定義されます。
Principal PはホストHから<kafka_resource> Rに対する<operation> Oを許可/拒否される

例えば、ユーザーJohnがホスト127.0.0.1からTopicコメントを表示できるようにルールを設定することができます。ホストは、Johnが接続しているマシンのIPアドレスです。




=== ACLの作成
このセクションでは、*ProducerユーザーとConsumerユーザーのそれぞれにACLを設定します。*

Producerユーザーには、Topic `secure-demo-again`への*書き込みは可能だが、作成や削除はできない* ようにACLを設定します。

Consumerユーザーには、Topic `secure-demo`からの*読み取りは可能だが、書き込みはできない* ようにACLを設定します。

* KafkaUserリソースの作成時にACLを定義できます。まずProducerユーザーのACLを作成しましょう。link:{openshift_cluster_console}[OpenShift console, window="console"]に移動し、*+* アイコンをクリックします。
+
image:common/create-yaml.png[]

* `Import YAML`オプションをクリックし、以下のYAMLをYAMLエディタに貼り付けます。
+
[source,yaml,role="execute",subs=attributes+]
----
apiVersion: v1
kind: Secret
metadata:
  name: producer-password
  namespace: kafka-{user_name}
type: Opaque
stringData:
  password: "producerpassword"
---
apiVersion: kafka.strimzi.io/v1beta2
kind: KafkaUser
metadata:
  name: producer
  namespace: kafka-{user_name}
  labels:
    strimzi.io/cluster: kafka
spec:
  authentication:
    type: scram-sha-512
    password:
      valueFrom:
        secretKeyRef:
          name: producer-password
          key: password
  authorization:
    type: simple
    acls:
      - resource:
          type: topic
          name: secure-demo
          patternType: literal
        operations:
          - Write
          - Describe
        host: "*"
----

* 上記のカスタムリソースは、ユーザー名 `producer`のKafkaUserを作成し、パスワードは`producer-password`という名前のSecretに保存されます。authorizationセクションでは、Topic `secure-demo`のACLを定義します。このユーザーはTopic `secure-demo-again`への書き込みは可能ですが、作成や削除はできません。createボタンをクリックしてください
+
image::m6/producer-user-yaml-create.png[]

* それでは、今作成したProducerユーザーを使用してTopicを削除してみましょう。

* *上部のターミナル* に移動し、まずProducerの認証情報を含む一時設定ファイルを作成します。
+
[source,sh,role="execute",subs=attributes+]
----
cat > /tmp/client.properties <<EOF
security.protocol=SASL_PLAINTEXT
sasl.mechanism=SCRAM-SHA-512
sasl.jaas.config=org.apache.kafka.common.security.scram.ScramLoginModule required username="producer" password="producerpassword";
EOF
----


* *上部のターミナル* に移動し、以下のコマンドを実行して、ProducerユーザーがTopic `secure-demo`を削除できるか確認します。
+
[source,sh,role="execute",subs=attributes+]
----
kafka-topics.sh --bootstrap-server kafka-kafka-bootstrap:9095 \
  --command-config /tmp/client.properties \
  --delete --topic secure-demo
----

* 以下のようなエラーメッセージが表示されるはずです。
+
[source,bash]
----
ERROR org.apache.kafka.common.errors.TopicAuthorizationException: Topic authorization failed.
----

* 上記のエラーは予想通りで、ProducerユーザーはTopicへの書き込み権限のみを持ち、削除権限を持たないため、ACLが動作していることを示しています。

* それでは、Topicへの書き込みを試してみましょう。*上部のターミナル* で以下のコマンドを実行し、Topicにいくつかのメッセージを書き込みます。
+
[source,sh,role="execute",subs=attributes+]
----
kafka-console-producer.sh \
  --bootstrap-server kafka-kafka-bootstrap:9095 \
  --producer.config /tmp/client.properties \
  --topic secure-demo
----

* 以下のようにいくつかのメッセージを書き込み、`ctrl+c`を押してProducerを停止します。
+
[source,bash]
----
> ACLs
> make
> your
> kafka
> cluster
> secure
----

* ACLが機能したことを確認するには、link:https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}/kafka[streams console, window="_amqstreams"] > topicsに移動し、Topic `secure-demo`をクリックしてメッセージを確認します。これにより、ProducerユーザーがTopic `secure-demo`への書き込みは可能だが、削除はできないことが示されます。
+
image::m6/secure-acl-message.png[]

* 次に、ConsumerユーザーのサンプルACLを実装しましょう。link:{openshift_cluster_console}[OpenShift console, window="console"]に移動し、*+* アイコン
+
image:common/create-yaml.png[] アイコンをクリックします。

* `Import YAML`オプションをクリックし、以下のYAMLをYAMLエディタに貼り付けます。
+
[source,yaml,role="execute",subs=attributes+]
----
apiVersion: v1
kind: Secret
metadata:
  name: consumer-password
  namespace: kafka-{user_name}
type: Opaque
stringData:
  password: "consumerpassword"
---
apiVersion: kafka.strimzi.io/v1beta2
kind: KafkaUser
metadata:
  name: consumer
  namespace: kafka-{user_name}
  labels:
    strimzi.io/cluster: kafka
spec:
  authentication:
    type: scram-sha-512
    password:
      valueFrom:
        secretKeyRef:
          name: consumer-password
          key: password
  authorization:
    type: simple
    acls:
      - resource:
          type: topic
          name: secure-demo
          patternType: literal
        operations:
          - Read
          - Describe
        host: '*'
      - resource:
          type: group
          name: console-consumer-
          patternType: prefix
        operations:
          - Read
        host: '*'
----

* このユーザーはTopic `secure-demo`への読み取りアクセスのみを持つことに注目してください。createボタンをクリックしてください。 

* このユーザーがTopicに書き込めるか再確認しましょう。このユーザーはConsumer用なので、Topicへの書き込みはできないはずです。

* まだProducerが実行中の場合は、ターミナルで`ctrl+c`を押して終了します。

* *上部のターミナル* に移動し、まずConsumerの認証情報を含む一時設定ファイルを作成します。
+
[source,sh,role="execute",subs=attributes+]
----
cat > /tmp/client.properties <<EOF
security.protocol=SASL_PLAINTEXT
sasl.mechanism=SCRAM-SHA-512
sasl.jaas.config=org.apache.kafka.common.security.scram.ScramLoginModule required username="consumer" password="consumerpassword";
EOF
----

* *上部のターミナル* で以下のコマンドを実行し、Consumerの認証情報を持つクライアントがTopicに書き込めるか(ACLでは許可されていない)確認します。
+
[source,sh,role="execute",subs=attributes+]
----
kafka-console-producer.sh \
  --bootstrap-server kafka-kafka-bootstrap:9095 \
  --producer.config /tmp/client.properties \
  --topic secure-demo
----

* Topicにメッセージを書き込んでみてください。
+
[source,bash]
----
> I
> am
> an
> evil
> consumer
> trying
> to
> write
> to
> a
> topic
----

* ConsumerユーザーがTopic `secure-demo`への書き込みが許可されていないことを示す、以下のようなエラーメッセージが表示されるはずです。
+
[source,bash]
----
2025-12-15 22:51:20,756] ERROR [Producer clientId=console-producer] Aborting producer batches due to fatal error (org.apache.kafka.clients.producer.internals.Sender)
org.apache.kafka.common.errors.ClusterAuthorizationException: Cluster authorization failed.
[2025-12-15 22:51:20,771] ERROR Error when sending message to topic secure-demo with key: null, value: 2 bytes with error: (org.apache.kafka.clients.producer.internals.ErrorLoggingCallback)
org.apache.kafka.common.errors.ClusterAuthorizationException: Cluster authorization failed.
pod "sinster-consumer" deleted
----



* `ctrl+c`を押して停止します。

* ACLが許可している、ConsumerユーザーがTopicを読み取れるか確認しましょう。*上部のターミナル* で以下のコマンドを実行します。
+
[source,sh,role="execute",subs=attributes+]
----
kafka-console-consumer.sh \
  --bootstrap-server kafka-kafka-bootstrap:9095 \
  --consumer.config /tmp/client.properties \
  --topic secure-demo \
  --from-beginning
----

* ProducerがTopic `secure-demo`に書き込んだすべてのメッセージが表示され、ConsumerユーザーがTopicを読み取ることができ、ACLが期待通りに実装されていることが示されます。

* `ctrl+c`を押してConsumerを停止します。

=== まとめ
ここまで見てきたのは、ACLの簡単な例です。企業は、既存のIDMからユーザーとグループを*Red Hat Single Sign-On* テクノロジーと同期して、細かいアクセス制御を実装できます。Red Hat Single Sign-OnでユーザーまたはグループのポリシーとACLルールを設定でき、Kafka管理者はこれを使用してACLによりKafkaクラスタをセキュアにできます。

Streams for Apache Kafkaは、Red Hat Single Sign-On Authorization Servicesを通じた*OAuth 2.0* トークンベースのAuthorizationの使用をサポートしており、セキュリティポリシーと権限を一元管理できます。

このトピックについてさらに学びたい場合は、このlink:https://developers.redhat.com/articles/2022/05/04/use-red-hats-sso-manage-kafka-broker-authorization#[ブログ記事,window="reference"]が良い出発点です。
