== Authentication
Kafka ListenerはAuthenticationを使用して、Kafkaクラスタへの安全なクライアント接続を確保します。Kafka Listenerは、基本的にKafkaクライアント(ProducerやConsumerなど)がKafka Brokerに接続するエントリーポイントです。ネットワークアドレス、ポート、プロトコル、認証方法など、クライアントがKafkaにどこから、どのように接続できるかを定義します。セキュリティ認証情報は、Cluster OperatorとUser Operatorによって作成および管理されます。{streams}は4つの主要な認証メカニズムをサポートしています。

* mTLS authentication (TLSによる暗号化が有効なListenerで使用)
* SASL SCRAM-SHA-512
* OAuth 2.0 token based authentication
* Custom authentication (Kafkaでサポート)

これまでのすべてのモジュールでは、認証なしでListenerを使用してきました。私たちのKafkaクラスタには、異なるプロトコルと認証方法を持つ複数のListenerがあります。

KafkaクラスタのListenerを簡単に見て、KafkaクラスタにおけるListenerの目的と設定について学びましょう。

* *下部のターミナル* で、以下のコマンドを使用してプロジェクトnamespace `kafka-{user_name}`にログインします。
+
[source,bash,role="execute",subs=attributes+]
----
oc project kafka-{user_name}
----


[IMPORTANT]
====
以下のようなエラーメッセージが表示された場合:

.出力
[source,textinfo,subs="attributes"]
----
error: You are not a member of project "kafka-user1".
You have one project on this server: showroom-mk2fc-1-user1
----
ログインしていることを確認するため、以下のコマンドを実行してください
[source,bash,role="execute",subs=attributes]
----
{login_command}
----
その後、下部のターミナルで `oc project kafka-{user_name}` コマンドを再実行してください。
====




* KafkaクラスタのListenerを確認するため、以下のコマンドを実行します。
+
[source,bash,role="execute",subs=attributes+]
----
echo "Ports:" && \
oc get svc kafka-kafka-bootstrap -o jsonpath='{range .spec.ports[*]}{.name}{" → "}{.port}{" ("}{.protocol}{")"}{"\n"}{end}' && \
echo -e "\nStrimzi Discovery:" && \
oc get svc kafka-kafka-bootstrap -o jsonpath='{.metadata.annotations.strimzi\.io/discovery}' | jq
----

* 以下のような出力が表示されるはずです。
+
[source,bash]
----
Ports:
tcp-replication → 9091 (TCP)
tcp-clients → 9092 (TCP)
tcp-clientstls → 9093 (TCP)
tcp-secure → 9095 (TCP)

Strimzi Discovery:
[
  {
    "port": 9092,
    "tls": false,
    "protocol": "kafka",
    "auth": "none"
  },
  {
    "port": 9093,
    "tls": true,
    "protocol": "kafka",
    "auth": "none"
  },
  {
    "port": 9095,
    "tls": false,
    "protocol": "kafka",
    "auth": "scram-sha-512"
  }
]
----

これが何を意味するのか簡単に解説しましょう。

* *Port 9091 – 内部Broker Replication: Kafka間のトラフィックに使用(BrokerがPartitionを相互にレプリケート)*
** 常に内部用で、クライアントには公開されません。
** Authenticationをサポートしていません。
** クラスタネットワーク内にのみ留まるべきです。
** BrokerとControllerへのアクセスを制限 — 外部やクライアントのPodがこれに到達すべきではありません。

* *Port 9092 – Plaintext Client Listener: セキュリティ保護されていないKafkaクライアントのデフォルトListener。これまでのモジュールで使用してきたListenerです。*
** TLSなし、Authenticationなし(auth: none)。
** ネットワークアクセスがあれば誰でもTopicの読み書きが可能です。
** 簡単なテストやローカル開発には便利ですが、本番環境では安全ではありません。
** NamespaceやClusterの外部に公開しないでください。

* *Port 9093 – TLS Client Listener: 暗号化されたTLSでのクライアント接続*

** 転送中の暗号化を提供(盗聴を防止)。

** デフォルトではAuthenticationなし(auth: none)のため、TLSクライアント証明書と組み合わせない限り匿名のまま。

** データの機密性のみが必要な場合、信頼できる内部アプリケーションには安全。

* *Port 9095 – SASL/SCRAM Secure Listener: セキュアで認証されたクライアントアクセス*

** SASL SCRAM-SHA-512経由でユーザー名/パスワードが必要(auth: scram-sha-512)。

** 認可されたユーザーのみが接続できることを保証。

** オプションでTLSでラップ可能(SASL_SSLになる)で、完全な暗号化+認証を実現。

** 一般的な本番環境セットアップに最適なバランス。


=== セキュリティの実践
それでは実際に見てみましょう。

* 以前のモジュールで作成したKafka toolboxコンテナにまだ接続していることを確認してください。
+
image::m6/tool-box-shell.png[]
+
[IMPORTANT]
====
toolboxコンテナに接続していない場合は、*上部のターミナル* で以下のコマンドを使用して再接続してください。

[source,sh,role="execute",subs=attributes]
----
oc rsh -n kafka-{user_name} kafka-toolbox
----

Module 1をスキップして以前にtoolboxコンテナを作成していない場合、 *Error from server (NotFound): pods "kafka-toolbox" not found* のようなエラーメッセージが表示される可能性があります。 *上部のターミナル* で以下のコマンドを使用して作成してください。

[source,sh,role="execute",subs=attributes]
----
oc run -n kafka-{user_name} kafka-toolbox \
  --image=registry.redhat.io/amq-streams/kafka-40-rhel9:3.0.0 \
  --restart=Never \
  --env="BOOTSTRAP=kafka-kafka-bootstrap.kafka-{user_name}.svc:9092" \
  --env="PATH=/opt/kafka/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin" \
  -- sleep infinity
----

数秒待ってから *`oc rsh -n kafka-{user_name} kafka-toolbox`* で接続してください


====



* **上部のターミナル** で、認証なしでセキュアな9095 Listenerを使用して新しいTopic `secure-demo`を作成します。以前のモジュールで正常に使用したのと同じコマンドを実行し、何が起こるか見てみましょう。
+
[source,sh,role="execute",subs=attributes+]
----
kafka-topics.sh --bootstrap-server kafka-kafka-bootstrap:9095 \
  --create --topic secure-demo --partitions 3 --replication-factor 1
----

* 1分程度待ち、Topic作成がタイムアウトしてコマンドが失敗することを確認してください。以下のエラーメッセージが表示されるはずです。
+
[source,bash]
----
Error while executing topic command : Timed out waiting for a node assignment. Call: createTopics
[2025-10-27 21:28:12,989] ERROR org.apache.kafka.common.errors.TimeoutException: Timed out waiting for a node assignment. Call: createTopics
 (org.apache.kafka.tools.TopicCommand)
pod "topic-create" deleted
pod kafka-user1/topic-create terminated (Error)
----

* これは、*認証なしでセキュアな9095 Listenerを使用している* ためです。

* このクラスタには、必要なすべての権限を持つ管理者ユーザーが既に作成されています。このユーザーを使用して、認証付きのセキュアなListenerでTopicを作成します。

* まず、必要な認証情報を含むローカル設定ファイルを作成しましょう。*上部のターミナル* で以下のコマンドを実行します。
+
[source,sh,role="execute",subs=attributes+]
----
cat > /tmp/client.properties <<EOF
security.protocol=SASL_PLAINTEXT
sasl.mechanism=SCRAM-SHA-512
sasl.jaas.config=org.apache.kafka.common.security.scram.ScramLoginModule required username="admin" password="adminpassword";
EOF
----


* それでは、正しい認証情報を使用して新しいTopicを再度作成しましょう。認証情報が設定ファイルの形式で渡され、認証付きのセキュアな9095 Listenerを使用していることに注目してください。
+
[source,sh,role="execute",subs=attributes+]
----
kafka-topics.sh --bootstrap-server kafka-kafka-bootstrap:9095 \
  --command-config /tmp/client.properties \
  --create --topic secure-demo --partitions 3 --replication-factor 1
----

* Topic `secure-demo`の作成が成功するはずです。https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}/kafka[streams console, window="console"] > topicsに移動して確認できます。


* リストにTopic `secure-demo`が表示されるはずです。これは、認証付きのセキュアな9095 Listenerを使用して作成したTopicです。
