== Kafka基礎 - Consumer GroupsとRebalancing
Kafka Consumer Groupsは、Kafka topicからのスケーラブルで高可用性のメッセージconsumption用に設計されています。各consumerはconsumer groupに属します。consumer groupはconsumerインスタンスのリストで、フォールトトレランス（耐障害性）とスケーラブルなメッセージ処理を保証します。consumer groupに1つのconsumerのみが含まれている場合、そのconsumerがすべてのpartitionのすべてのメッセージを処理する責任を持ちます。 

複数のconsumerが同じgroupに参加してtopicにサブスクライブすると、Kafkaはtopicのpartitionをそれらの間で分散し、各partitionがそのgroupの1つのメンバーのみによってconsumeされることを保証します。これにより、メッセージ処理の水平スケーリングが可能になります。groupにより多くのconsumerを追加することで、ワークロードを効率的に分散でき、アプリケーションが大量のデータを処理できるようになります。

*topicのpartition数よりも多くのconsumerをconsumer groupに追加すると、余分なconsumerはメッセージを受信せずにアイドル状態のままになります。*

image::m1/consumer-group-architecture.png[]

group内のconsumerへのpartitionの動的な割り当ては、*rebalancing* と呼ばれるプロセスを通じて管理されます。Rebalanceは、consumerがgroupに参加または離脱したとき、クラッシュしたとき、またはサブスクライブしているtopicのpartitionが変更されたときに発生します。

image::m1/cg-rebalance.png[]

実際にどのように動作するか見てみましょう。

* 再度、*上部のターミナル* でkey-valueペアのメッセージを生成するproducerを実行します。 
+
[source,sh,role="execute",subs=attributes+]
----
i=1; while true; do echo "k$((i%10)):msg-$i"; i=$((i+1)); sleep 0.3; done | \
  kafka-console-producer.sh \
  --bootstrap-server $BOOTSTRAP \
  --topic demo \
  --property parse.key=true \
  --property key.separator=:
----

* それでは、簡単なbashスクリプトを使用して、同じconsumer group *demo-cg* で3つのconsumerを1つずつ起動しましょう。スクリプトを進めて新しいconsumerを起動し続けると、consumer間でpartitionのrebalancingが発生することがわかります。

* toolboxコンテナ内にいる場合は、*下部のターミナル* でシェルを終了します。
+
[source,bash,role="execute",subs=attributes+]
----
exit
----

* *下部のターミナル* で、`kafka-{user_name}`プロジェクトにいることを確認します。
+
[source,bash,role="execute",subs=attributes+]
----
oc project kafka-{user_name}
----


* まず、*下部のターミナル* で以下のコマンドを実行して最初のconsumerを起動します。*2つ目のconsumerを起動するよう促されても、まだEnterを押さないでください。* 
+
[source,bash,role="execute",subs=attributes+]
----
curl -s https://raw.githubusercontent.com/streams-kafka-workshop/workshop_scripts/refs/heads/main/demo-cg_consumers.sh | bash
----

* https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}/kafka[streams console, window="_amqstreams"] > consumer groups > demo-cgに移動して、最初のconsumerと各consumerに割り当てられているpartitionを確認します。
+
image::m1/demo-cg.png[]

* 現在1つのconsumer（cg-c1）しかないため、topic `demo`のすべてのpartitionが割り当てられます。すべてのpartitionに送信されたすべてのメッセージは、この単一のconsumer（cg-c1）によってconsumeされます。
+
image::m1/demo-cg-c1.png[]




* すべてのpartitionがcg-c1に割り当てられていることを確認したら、**下部のターミナル** でEnterを押して2つ目のconsumer（cg-c2）を起動します。*まだ3つ目のconsumerを起動するためにEnterを押さないでください。*

* https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}/kafka[streams console, window="_amqstreams"]に戻って、cg-c1とcg-c2に割り当てられているpartitionを確認します。以下に示すように、rebalanceが発生し、partitionがcg-c1とcg-c2の間で再割り当てされることがわかります。


* 1つのconsumerが1つのpartitionを担当し、もう1つのconsumerが他の2つのpartitionを担当していることを確認できます。
+
image::m1/demo-cg-c1-c2.png[]

* *下部のターミナル* でEnterを押して3つ目のconsumer（cg-c3）を起動します。別のrebalanceが発生し、partitionがcg-c1、cg-c2、cg-c3の間で再割り当てされることがわかります。*まだ4つ目のconsumerを起動するためにEnterを押さないでください。*

* https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}/kafka[streams console, window="_amqstreams"]に戻って、cg-c1、cg-c2、cg-c3に割り当てられているpartitionを確認します。
+
image::m1/demo-cg-c1-c2-c3.png[]


* *下部のターミナル* でEnterを押して4つ目のconsumerを起動してください。まだEnterを押して他のconsumerを停止しないでください。

* 以下に示すように、4つ目のconsumer（cg-c4）が起動されますが、partitionは割り当てられず、アイドル状態のままになることがわかります。これは、3つのpartitionと3つのconsumerがあるためです。partitionが残っていないため、4つ目のconsumerにはpartitionが割り当てられません。Kafkaは、メッセージの重複処理を避けるために、任意の時点で各partitionがconsumer groupの1つのメンバーのみによってconsumeされることを保証します。
+
image::m1/demo-cg-c1-c2-c3-c4.png[]

=== OffsetsとLag
*Offsets:* すべてのメッセージは、partition内で一意の連続番号を取得します。この「ページ番号」により、consumerは読み取りを停止した正確な位置をマークできます。

*Lag:* Lagは、consumerがどれだけ遅れているかを測定します。

例：brokerがメッセージ#100を書き込んでいるが、consumerが現在メッセージ#90を読み取っている場合、Lagは10です。

高いlagは、アプリケーションが遅く、受信データに追いつくのに苦労していることを意味します。

image::m1/roundup-image.png[]



* https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}/kafka[streams console, window="_amqstreams"] > topics > consumer groups > demo-cgに移動し、前に起動したconsumer cg-c1を展開します。

* producerが継続的にメッセージを生成し続けると、以下に示すように、各consumerのlag、committed offset、end offsetが変化し続けることを確認します。lagは基本的に *End Offset - Committed Offset* です。lag、committed offset、end offsetの値が変化していない場合は、ページを更新してください。
+
image::m1/lag-offset.png[]


* 複数のconsumerがある場合、Consumer group demo-cgは、以下に示すように、各consumerのlagの合計であるgroup全体の全体的なlagも維持します。
+
image::m1/group-lag.png[]

* 最初のconsumer（cg-c1）以外のすべてのconsumerを停止すると、すべてのpartitionが自動的にcg-c1に再割り当てされます。Enterを押してconsumer cg-c2、cg-c3、cg-c4を停止してください。rebalancingが発生し、すべてのpartitionが自動的にcg-c1に再割り当てされることがわかります。*次のセクションで必要になるため、まだEnterを押して最初のconsumer（cg-c1）を停止しないでください。*
+
image::m1/demo-cd-c1-only.png[]

* それでは、*上部のターミナル* でproducerを停止し（Ctrl+C）、*下部のターミナル* でEnterを押してすべてのconsumerも停止してください。


 





// === Summary
// * A consumer group is a coordinated set of consumers that share a topic’s partitions.

// * At most one consumer per partition per group: if your topic has 3 partitions, up to 3 consumers in the same group can read in parallel; a 4th will sit idle.

// * Kafka tracks the group’s offset per partition (so the group, as a whole, has a reading position).

// * A different group has its own offsets, so it can re-read the entire topic independently.
