== シンプルなConsumerアプリケーションの実行
前のセクションで分析したシンプルなストリーミングアプリケーションを実行して、consumerが実際に動作することを確認しましょう。

* Devspaces IDEに戻り、`clear`と入力してEnterキーを押してターミナルウィンドウをクリアします。
+
[source,bash,role="execute",subs=attributes+]
----
clear
----

* ターミナルでインタラクティブアプリケーションを実行します。
+
[source,bash,role="execute",subs=attributes+]
----
./run.sh
----

* 数秒でアプリケーションがビルドされ、オプションのメニューが表示されます。
+

[source,bash]
----
========================================
   Kafka Java Tutorial Application
========================================

Main Menu:
1. Run Producer
2. Run Consumer
3. Exit

Enter your choice:
----

* オプション2を選択してconsumerを実行します。オプション2を入力してEnterキーを押します。

* 最初に、`auto.offset.reset`の値を入力するように求められます

* `latest`と入力してEnterキーを押します。
+
NOTE: このプロパティは、consumerまたはconsumer groupに初期offsetが存在しない場合、またはoffsetが範囲外の場合にconsumerがどうすべきかを定義します* *"earliest"* = topicの最初から開始(最初からすべてのメッセージを読む) *"latest"* = 終わりから開始(consumer開始後に送信された新しいメッセージのみを読む)

* 次のプロンプトで`fetch.min.bytes`を求められます。Enterキーを押してデフォルト値を使用します。

* 次のプロンプトで`fetch.max.wait.ms`を求められます。Enterキーを押してデフォルト値を使用します。

* 次のプロンプトで`Enter number of messages`を求められます。以下の値を入力してEnterキーを押します。
+
[source,bash,role="execute",subs=attributes+]
----
30
----

* consumerがproducerが送信したばかりのメッセージを読み取らないことを確認してください。これは、`auto.offset.reset`に`latest`値を選択したため、consumerがtopicの終わりから開始し、consumer開始前に送信されたメッセージは消費しないためです。streams consoleで再確認することで同じことを確認できます。

* link:https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}/kafka/[streams console, window="_amqstreams"] > consumer groupsに移動し、*tutorial-consumer-group* をクリックします
+
image::m2/console_tutorial-consumer-group.png[]

* consumer groupの全体的なlagが*topicに未読メッセージがあったにもかかわらずゼロ* であることを確認してください。これは、consumerがtopicの終わりから読み取っており、committed offsetの値がend offsetと同じ値に設定されているため、lagがゼロになるためです。
+
image::m2/console_consumer-lag-zero.png[]


* Enterキーを押して再度`1`を押し、producerを再度実行します。すべての*デフォルト値を受け入れてEnterキーを押します* 。メッセージが送信されることを確認してください。

* それではEnterキーを押して再度`2`を押し、consumerを再度実行します。今回は、プロンプトでEnterキーを押して`auto.offset.reset`、`fetch.min.bytes`、`fetch.max.wait.ms`の*デフォルト値* を選択しましょう。

* 'Enter number of messages'プロンプトの場合。以下の値を入力してEnterキーを押します。
+
[source,bash,role="execute",subs=attributes+]
----
30
----


* consumerが実行され、producerによって送信されたばかりのすべてのメッセージを消費することが確認できます。メッセージはターミナルに表示されます。


=== オプション演習:fetch.min.bytesとfetch.max.wait.msで実験する
`fetch.min.bytes`と`fetch.max.wait.ms`の異なる値で実験して、consumerの動作にどのように影響するかを確認することもできます。

* Enterキーを押して再度`1`を押し、producerを再度実行します。すべてのデフォルト値を受け入れてEnterキーを押します。

* それではEnterキーを押して`2`を押し、再度Enterキーを押してconsumerを再度実行します。

* 今回は、`fetch.min.bytes`プロンプトに対して`1024`を入力してEnterキーを押します。

* `fetch.max.wait.ms`プロンプトに対して`4000`を入力してEnterキーを押します。

* この例ではバイト数が1024バイトを超えることはないため、`fetch.min.bytes`は無視され、consumerはメッセージをフェッチする前に少なくとも4秒待機します。メッセージがターミナルに表示される前に約4〜6秒の遅延が確認できるはずです。

* *Enter number of messages* プロンプトの場合。以下の値を入力してEnterキーを押します。メッセージが表示される前の遅延を確認できます。
+
[source,bash,role="execute",subs=attributes+]
----
30
----



=== クリーンアップ
* ターミナルウィンドウで *ctrl+c* を押してアプリケーションを終了することを確認してください。

=== まとめ
このセクションが、consumer APIの高レベルの理解を得て、いくつかのconsumer設定が実際に動作することを確認するのに役立ったことを願います。アプリケーションの他のオプションを自由に試して、consumerの動作を観察してください。
