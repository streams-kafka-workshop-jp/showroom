== Debezium Connectorのデプロイ

まず、Debezium Connectorをデプロイしましょう。{streams}では、Kafka Connect connectorはKubernetes Custom Resourceとしてデプロイでき、streams for Apache Kafka operatorによって処理されます。

connectorをデプロイするために、OpenShift Dev Spacesを使用します。OpenShift Dev SpacesはKubernetesとコンテナを使用して、ブラウザウィンドウからアクセス可能な、一貫性があり安全でゼロ設定の開発環境を提供します。

* ブラウザウィンドウで、OpenShiftクラスターコンソールを指しているブラウザタブに移動します。コンソールでブラウザタブを開いていない場合は、link:{openshift_cluster_console}[OpenShift console, window="console"]に移動してコンソールを起動します。必要に応じて、ユーザー名とパスワード（`{user_name}`/`{password}`）でログインします。

* まだDevspacesを起動していない場合は、OpenShiftコンソールの右上隅からDevspaces IDEを開きます。
+
image::m4/devspaces.png[]

* "log in with OpenShift" ボタンをクリックします。
+
image::m4/login-openshift.png[]

* 認証情報を使用してログインします - `{user_name}` と `{password}`。


* Devspacesダッシュボードから、*open* ボタンをクリックしてワークスペースを起動します。
+
image::m4/devspaces-launch.png[]

* ワークスペースが起動したら、 *Yes, I trust the authors* をクリックして続行します。
+
image::m4/devspaces-trust.png[]


* ワークスペースには、Debezium connectorを含む、ワークショップ中に使用するすべてのリソースが含まれています。ワークスペースの左側にあるプロジェクトエクスプローラーで、*kafka-workshop/module-cdc* フォルダに移動し、*debezium-connector.yaml* ファイルを開きます。
+
image::m4/devspaces-workspace-debezium-connector.png[]

* *debezium-connector.yaml* ファイルはDebezium Connectorを記述しています。このファイルには、connectorがターゲットデータベーステーブルの変更をキャプチャするために知っておく必要があるすべての詳細が含まれています。いくつかの重要な設定の詳細:
** *class*: Debezium connectorの実装クラス。ソースデータベースとしてPostgreSQLを使用しているため、適切なconnectorは *io.debezium.connector.postgresql.PostgresConnector* です。
** *plugin.name*: Debezium connectorはPostgreSQLトランザクションログから読み取るための異なるメカニズムをサポートしています。*pgoutput* はPostgreSQL 10以降の標準的な論理デコーディング出力プラグインです。
** *+database.*+*: データベースの接続詳細。PostgreSQLは、トランザクションログから読み取るために必要な権限を持つ特定のユーザー（*debezium*）で設定されていることに注意してください。
** *topic.prefix*: Debezium change eventを受信するKafkaトピックのプレフィックス。トピックのフルネームは *<prefix>.<schema>.<table>* です。
** *schema.include.list*: Change Data Captureプロセスに含めるスキーマ。
** *table.include.list*: 含めるテーブルの名前。このユースケースでは、*customer*、*orders*、*line_item* テーブルに関心があります。

* connectorをデプロイする前に、データベースホスト名のプレースホルダーを実際の値に置き換える必要があります。 *14行目で、置き換えます*
+
----
<REPLACE WITH DATABASE HOSTNAME>
----
+
以下に
+
[source,textinfo,role=execute,subs="attributes"]
----
globex-db.globex-{user_name}.svc.cluster.local
----
+
これはGlobex retailアプリケーションデータベースの内部DNS名です。

* 数秒後、Debezium connectorはPostgreSQLデータベースの *customer*、*orders*、*line_item* テーブルの変更の監視を開始し、検出された各変更に対して対応するKafkaトピックにchange eventを生成します。

* connectorが期待通りに動作していることを確認する1つの方法は、change eventを受信するKafkaトピックをチェックすることです。

* https://amq-streams-console-{user_name}.{openshift_cluster_subdomain}[streams for Apache Kafka console, window="_amqstreams"]に移動します。

** これにより、*streams for Apache Kafka console* のログインページにリダイレクトされます。

** このワークショップの目的のため、まだサインインしていない場合は *Click to login anonymously* を選択してコンソールにアクセスします。
+
image::common/console_anonymous.png[]

* 左側の *Topics* メニューに移動します。
+
`globex.updates` という用語でトピックを *Name* でフィルタリングします。最初は、change eventを受信する1つのトピック `globex.updates.public.customer` のみが表示されます。
+
image::m4/console-cdc-filter-topics.png[]

* Globexアプリケーションデータベースには *customer* テーブルに数百人の顧客のレコードが含まれているため、これらの各レコードに対してchange eventが期待されます。streams for Apache Kafka consoleのトピックページで、*globex.updates.public.customer* トピックをクリックします。これにより、トピックの詳細を含むビューが開きます。最上位（最新）メッセージのOffsetが _199_ であることに注目してください。これは *customer* テーブルのレコード数に対応しています。
+
NOTE: Offsetは _0_ から始まります。Offsetが _199_ の場合、トピックには _200_ 個のメッセージがあることを意味します。
+
image::m4/amqconsole-debezium-topic-customers.png[]

* すべてのメッセージを展開してその内容を調べることができます。この場合、各メッセージの本文はJSON形式のDebezium change eventで構成されています。最上位のメッセージをクリックして展開します。
+
image::m4/amqconsole-debezium-topic-customers-200.png[]

* Debezium change eventには明確に定義された構造があります。次の要素に特に注意してください:
** *before*: トランザクション前のレコードの状態。change eventは新しく読み取られたレコードに対応するため、前の状態はありません。
** *after*: トランザクション後のレコードの状態。これはデータベース内のレコードの現在の状態のJSON表現です（テーブルのすべての列がJSONフィールドになります）。
** *op*: change eventにつながる操作。可能な値は _create_ の場合は '*c*'、_update_ の場合は '*u*'、_delete_ の場合は '*d*'、_read_ の場合は '*r*' です。Debezium connectorがデプロイされたときに *customer* のレコードはすでに存在していたため、操作は '*r*' です。

* Globexアプリケーションデータベースには現時点で注文情報が含まれていないため、*globex.updates.public.orders* と *globex.updates.public.line_item* トピックはまだ作成されていません。これはstreams for Apache Kafka consoleを通じて確認できます。
